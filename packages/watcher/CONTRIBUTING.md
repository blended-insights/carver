# Contributing to Carver Watcher

Thank you for your interest in contributing to the Carver Watcher service! This document provides guidelines and instructions for contributing to this project.

## Code of Conduct

Please be respectful and constructive in all interactions related to this project. Harassment or disrespectful behavior will not be tolerated.

## How to Contribute

### Reporting Issues

If you find a bug or have a suggestion for improvement:

1. Check if the issue already exists in the repository's issue tracker.
2. If not, create a new issue with:
   - A clear, descriptive title
   - A detailed description of the issue or suggestion
   - Steps to reproduce (for bugs)
   - Expected and actual behavior (for bugs)
   - Version information

### Pull Requests

We welcome contributions through pull requests:

1. Fork the repository
2. Create a new branch for your changes
3. Make your changes, following our coding standards
4. Add or update tests for your changes
5. Update documentation as needed
6. Submit a pull request with a clear description of the changes

### Development Process

1. Set up your development environment as described in DEVELOPMENT.md
2. Make your changes in a feature branch
3. Ensure all tests pass
4. Push your changes and create a pull request

## Coding Standards

### TypeScript Guidelines

- Use TypeScript for all new code
- Follow the existing code style
- Use meaningful variable and function names
- Add JSDoc comments for public APIs
- Avoid `any` types when possible
- Use interfaces and type declarations

### Testing

- Write unit tests for all new functionality
- Maintain or improve test coverage
- Tests should be independent and focused

### Documentation

- Update README.md for user-facing changes
- Update DEVELOPMENT.md for developer-facing changes
- Add comments to complex code sections
- Document public APIs with JSDoc

## Commit Messages

Follow these guidelines for commit messages:

- Use the present tense ("Add feature" not "Added feature")
- Use the imperative mood ("Move cursor to..." not "Moves cursor to...")
- Limit the first line to 72 characters or less
- Reference issues and pull requests after the first line
- Structure commits logically (don't mix unrelated changes)

## Branch Naming

Use this format for branch names:

- `feature/short-description` - For new features
- `fix/issue-description` - For bug fixes
- `docs/description` - For documentation updates
- `refactor/description` - For code refactoring

## Review Process

All submissions require review:

1. A maintainer will review your pull request
2. Changes may be requested before merging
3. Once approved, a maintainer will merge your pull request
